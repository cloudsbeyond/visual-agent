# Tier 1 - Tool-Calling Adapter

Exposes L3 Scenario intents to a host tool/function-calling boundary. This adapter is a **thin
transport mapping only**: it maps one host tool call to one `task_flow execute-json` invocation.
The host owns observation interpretation, decision, approval, and task success judgement. This
adapter never calls a model, chooses a tool, plans autonomously, or judges task success. See
`../../integration/host-embedding.md` section 6 (tool-calling schema) and section 3
(invocation model).

## Two pieces

1. **Tool definitions** - generated by the runtime, not by this adapter:

   ```bash
   swift ../../runtimes/desktop-gui/src/task_flow.swift --manifest <manifest.json> tools-schema
   # or, if built:  ../../runtimes/desktop-gui/bin/task_flow --manifest <manifest.json> tools-schema
   ```

   Emits `{ schemaVersion, scene, tools: [ { name, description, parameters(JSON Schema) } ] }`.
   `scene` is the current desktop GUI manifest artifact name. Register each `tool` with your
   host's tool/function-calling API. `name` is the Scenario intent name.

2. **Invocation mapping** - `invoke.mjs` translates one tool call into one `execute-json`:

   ```bash
   node invoke.mjs --manifest <manifest.json> [--state <path>] '{"name":"<intent>","arguments":{...}}'
   echo '{"name":"<intent>","arguments":{...}}' | node invoke.mjs --manifest <manifest.json> -
   ```

   It builds `{ intent:name, ...arguments }`, runs `task_flow execute-json`, and passes the runtime
   JSON straight back, including the integration section 4 error envelope on failure.

## Contract For The Host

- The host decides which tool to call and with what arguments. It runs or coordinates the visual
  model. This adapter never calls a model, chooses a tool, or judges task success. (INT section 2)
- Branch on the response: `kind:"action_result"` (dispatched) vs `kind:"error"` + `reason` (failed).
  Remember `ok:true` means *dispatched*, not *succeeded* - verify by a follow-up observation.
  (OVERVIEW section 4 `dispatched-not-succeeded`)
- For concurrency, pass a per-session `--state <path>`. (INT section 5)
- For dry validation without touching the UI, call `task_flow preview-json` directly instead of
  this adapter.

## Example (dryRun, placeholder reference scenario)

```bash
node invoke.mjs \
  --manifest ../../runtimes/desktop-gui/examples/reference-surface/manifest.json \
  '{"name":"closeOverlay","arguments":{"dryRun":true}}'
```

Expected: a `kind:"action_result"` JSON with `dryRun:true` results.

## Runtime resolution

`invoke.mjs` prefers a compiled `../../runtimes/desktop-gui/bin/task_flow` when present (INT
section 9), otherwise falls back to `swift ../../runtimes/desktop-gui/src/task_flow.swift`.
Build binaries with `bash ../../runtimes/desktop-gui/scripts/build.sh`.
