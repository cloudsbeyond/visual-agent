# visual-agent

我在构建 **visual-agent**：一套协议优先的框架，用于让视觉模型宿主安全操作软件界面和简单物理面板。

视觉模型可以看懂界面，也可以推理可见状态。但这还不足以让操作变得可靠。宿主需要一套结构，明确运行时能观察什么、能执行什么，具体交互界面意味着什么，哪些动作安全，以及成功如何被验证。

这套结构就是 **visual-agent**。

## 问题

很多计算机使用、图形界面自动化、设备控制和简单具身控制项目，都会从智能体循环、沙箱、浏览器、控制器或机器人栈开始。这些都重要，但它们中间缺了一层：从视觉理解到可信操作之间，需要协议和交互界面说明体系。

我要补的就是这一层：

- 在智能体、规划器和模型适配器之下；
- 在截图、摄像头、输入事件、设备命令和执行后端之上；
- 不绑定模型供应商；
- 不绑定具体软件、网站、设备或控制界面；
- 明确描述安全、状态和验证。

## 可操作交互界面

**Operational Surface（可操作交互界面）** 指一个有明确边界、可以被观察和操作的界面，它可以被：

- 视觉模型或宿主观察；
- 通过声明式运行时能力执行动作；
- 用对应的交互界面说明书描述；
- 在每次关键动作后通过新的观察验证。

它可以是软件 GUI、浏览器页面、移动界面、原生应用窗口、打印机或 ATM 面板、家电控制面板、仪表显示，也可以是简单具身控制中的按钮、旋钮、指示灯和设备屏幕组合。核心协议处理的是“可见交互界面的操作”这一层；完整机器人、导航、抓取、力控和世界建模不属于 P0。

这就是协议的 **L0：Operational Surface Contract**。L0 定义什么样的可见交互界面可以进入 visual-agent 循环，先于任何运行时动作、交互界面说明书或场景清单。

## 运行模型

智能循环由外部宿主负责：

```text
observe -> decide -> act -> verify
```

这个项目负责这个循环周围的契约：

- 什么可操作交互界面有资格进入这个循环；
- 运行时或适配器能观察什么、执行什么；
- 一个具体可操作交互界面如何被描述；
- 一个任务如何变成安全动作计划；
- 结果和失败如何表达；
- 为什么动作已派发不等于任务成功。

## 四层结构

| 层 | 说明书 / 契约 | 目的 |
|---|---|---|
| L0 | 可操作交互界面契约（Operational Surface Contract） | 定义什么算可操作交互界面、界面边界、观察参考系、安全包络和最小验证预期。 |
| L1 | 运行时能力手册（Runtime Capability Manual） | 描述某类运行时或适配器能观察什么、执行什么，并返回机器可读结果。 |
| L2 | 交互界面说明书（Surface Manual） | 描述一个具体交互界面：可观察状态、安全目标或控件、动作边界和验证规则。 |
| L3 | 场景需求（Scenario Requirements） | 描述具体任务意图和场景清单，并展开为安全的运行时动作计划。 |

四层必须分开。L0 定义交互界面资格和边界。L1 不理解界面语义。L2 不重新实现运行时机制。L3 不直接发出鼠标、设备或执行器事件。宿主负责判断，运行时负责执行，成功必须来自新的观察。

## 我不做什么

我不把这个项目做成：

- computer-use 智能体；
- 规划器；
- 模型适配器；
- 以 MCP 服务为核心的产品；
- 通用重型浏览器自动化框架；
- 机器人框架或完整具身智能栈；
- 把具体应用、网站或设备脚本写进通用运行时代码的自动化集合。

这些以后可以围绕项目出现，但不能定义核心。**visual-agent** 这个名字表达的是“视觉模型操作可见交互界面”的概念，不是一个自主智能体产品。

## 核心契约

P0 定义为：

- 面向 visual-agent 的协议优先 L0/L1/L2/L3 交互界面、运行时和说明体系；
- 模型无关、交互界面无关的核心；
- 交互界面专属知识不进入通用运行时代码；
- `ActionResult.ok == true` 只表示动作已派发或在运行时边界完成，不表示任务成功；
- 任务成功只能通过新的观察验证；
- 失败必须机器可读；
- 安全审批必须显式发生。

当前 `runtimes/desktop-gui` 是第一个桌面 GUI 参考运行时。它用软件界面证明这套契约，但不是产品边界。浏览器、移动端、物理面板和具身控制运行时以后可以作为适配器或拆分项目继承同一套 L0/L1/L2/L3 契约。
